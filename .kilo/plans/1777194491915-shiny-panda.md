# Plan: User Login Feature (POST /api/users/login)

## Overview
Implement user login functionality with session management using UUID tokens stored in a sessions table.

## Current State
- Bun + Elysia + Drizzle + MySQL project already set up
- User registration (POST /api/users) already implemented
- bcryptjs already installed for password hashing
- Users table has: id, name, email, password, created_at

## Tasks to Complete

### 1. Database Schema - Add Sessions Table
**File:** src/db/schema.ts
**Actions:**
- Add sessions table definition using Drizzle MySQL schema
- Fields: id (serial, PK), token (varchar 255, not null), user_id (integer, FK to users.id), created_at (timestamp, default now)
- Add foreign key relationship to users table

### 2. Database Migration
**Actions:**
- Run bun run db:generate to create migration file
- Apply migration to database using bun run db:migrate (may need manual SQL if password validation fails)

### 3. Update Service Layer
**File:** src/service/users-service.ts
**Actions:**
- Create interface for login input (email, password)
- Implement loginUser() function:
  - Query users table by email
  - If no user found: throw Error('Email atau password salah')
  - Compare password with bcrypt.compare()
  - If mismatch: throw same Error('Email atau password salah')
  - Generate UUID token using crypto.randomUUID()
  - Insert session record into sessions table
  - Return token object: { token: 'uuid-string' }

### 4. Update Route Layer
**File:** src/routes/users-route.ts
**Actions:**
- Add POST /api/users/login endpoint
- Elysia schema validation:
  - email: t.String({ format: 'email' }) - required
  - password: t.String() - required
- Call loginUser() from service
- Error handling:
  - 401 for login errors (email/password incorrect)
  - 422 for validation errors (automatic)
- Success response: 200 with { data: { token: '...' } }

### 5. Testing
**Actions:**
- Start dev server
- Test valid login: 200 + token
- Test invalid email: 401
- Test invalid password: 401 (same message)
- Test missing fields: 422
- Verify session created in database
- Verify token is valid UUID format
- Verify no sensitive data leaked in responses

## Technical Considerations

### Security
- Same error message for both email not found and wrong password
- UUID tokens (not JWT) stored in sessions table
- Password comparison uses bcrypt.compare() (timing-safe)
- Plain text passwords never stored or logged

### Architecture
- Service layer: pure business logic (no HTTP concerns)
- Route layer: HTTP handling, validation, status codes
- Database layer: Drizzle ORM with query builder

### Error Handling
- Service throws Error with specific messages
- Route catches and converts to appropriate HTTP status codes
- 401 for authentication failures
- 422 for validation failures

## Acceptance Criteria
- [ ] Sessions table created via migration
- [ ] POST /api/users/login returns 200 + token for valid credentials
- [ ] POST /api/users/login returns 401 + same error message for invalid email/password
- [ ] POST /api/users/login returns 422 for invalid request body
- [ ] UUID token stored in sessions table
- [ ] User can have multiple sessions
- [ ] No sensitive data in responses
